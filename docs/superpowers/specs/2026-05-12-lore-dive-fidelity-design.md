# lore-dive Session Fidelity 设计文档

**日期：** 2026-05-12
**问题：** session 文件记录的是压缩摘要，丢失了对话中的 ASCII 图、表格、详细解释
**目标：** session 文件内容与对话输出完全一致，zero-loss

---

## 问题根因

当前 skill 的阶段 2 流程是「先回答用户，再写入 session」。AI 在写入时存在天然的压缩倾向——已经说过的内容，重新记录时会无意识地概括。即使指令写着「完整回答」，执行时仍会丢失图表和细节。

---

## 设计方案：先写后读（Write-First Protocol）

### 核心原则

**session 文件是唯一信源。** AI 先将完整回答写入 session，再将文件内容输出给用户。用户看到的 = 文件里存的，物理上是同一份，从根本上消除压缩动机。

### 阶段 2 新流程（替换原有步骤 1-2）

**原流程：**
```
1. 回答用户的问题
2. 读取 session → 追加压缩版 → 写入 session
```

**新流程：**
```
1. 构思完整回答（含所有 ASCII 图、表格、代码块，不截断）
2. Read session 文件当前内容
3. Write session 文件：在末尾追加完整回答块
4. 将刚才写入的内容原文输出给用户（不重新生成，不改写）
5. （如需）判断分支提示，追加到 session 并同步输出
```

### session 写入格式（不变）

```markdown
## Q<n> <简短标题>

**问：** <用户原文>

**答：** <完整回答，含所有 ASCII 图、表格、代码块，与输出给用户的内容字符级别一致>
```

### 对其他阶段的影响

- **阶段 0（Session Recovery）**：不变
- **阶段 1（新建 Session）**：不变，根问题的首轮回答也遵循新流程
- **阶段 3（生成文档）**：不变，但因 session 保真，合成文章可直接复用图表和详细内容，质量更高

---

## 实现要点

1. **指令措辞**：明确说「先写入 session，再将写入内容输出给用户」，而不是「写入完整回答」——前者是操作顺序约束，后者是内容质量约束，前者更强
2. **禁止改写**：输出给用户的内容必须来自 session 文件的刚写入部分，不允许「重新生成一遍」
3. **分支提示**也遵循同样协议：先追加到 session，再输出

---

## 不在本次范围内

- 阶段 3 合成文章对图表的特殊处理策略
