# TCP 连接的建立与关闭：三次握手与四次挥手

TCP（Transmission Control Protocol）是互联网最核心的传输层协议之一，它提供面向连接、可靠、有序的字节流传输。每一个 TCP 连接从建立到关闭，都经历了严格定义的握手过程——建立连接用**三次握手**，关闭连接用**四次挥手**。理解这两个过程，是理解 TCP 可靠性保证的基础。

---

## 一、前置概念：TCP 报头关键字段

在理解握手与挥手之前，需要先认识 TCP 报头中几个核心字段：

- **SEQ（Sequence Number）**：当前报文的序列号，用于标识字节流中的位置
- **ACK（Acknowledgment Number）**：期望收到对方下一个字节的序列号
- **SYN 标志位**：用于连接建立阶段，Synchronize 的缩写
- **FIN 标志位**：用于连接关闭阶段，表示「我这边没有数据要发了」
- **ACK 标志位**：表示 ACK 字段有效

其中，**初始序列号（ISN, Initial Sequence Number）** 由操作系统随机生成。随机化的目的是防止旧连接残留的报文被新连接误认，也防止序列号被预测攻击。

SYN 和 FIN 报文各自占一个序列号，这是为什么确认号总是对方序列号加一。

## 二、三次握手：建立连接

### 目标

三次握手的核心目标有两个：
1. **双方互相确认各自的发送和接收能力均正常**
2. **同步双方的初始序列号（ISN）**

### 过程

```
客户端 (Client)                          服务端 (Server)
     |                                        |
     |  1. SYN  SEQ=x                         |
     | -------------------------------------> |  LISTEN → SYN_RCVD
     |                                        |
     |  2. SYN-ACK  SEQ=y, ACK=x+1           |
     | <------------------------------------- |
     |  SYN_SENT → ESTABLISHED                |
     |                                        |
     |  3. ACK  SEQ=x+1, ACK=y+1             |
     | -------------------------------------> |  SYN_RCVD → ESTABLISHED
```

**第一次握手（SYN）**：客户端发送 SYN 报文，携带自己的初始序列号 x，进入 `SYN_SENT` 状态。告诉服务端「我想连接你，我的初始序列号是 x」。

**第二次握手（SYN-ACK）**：服务端收到后，回复 SYN-ACK 报文，携带自己的初始序列号 y，同时 `ACK = x+1` 表示已收到 x。服务端进入 `SYN_RCVD` 状态。

**第三次握手（ACK）**：客户端收到 SYN-ACK 后进入 `ESTABLISHED`，回复 ACK，`ACK = y+1` 确认收到了服务端的序列号。服务端收到后也进入 `ESTABLISHED`，连接建立完成。

### 为什么不能是两次？

两次握手有一个致命漏洞：**服务端无法知道客户端是否收到了自己的 ISN**。更严重的是，若一个旧的 SYN 报文在网络中滞留后延迟到达服务端，服务端会以为是新连接请求，直接进入 ESTABLISHED 并分配资源——但客户端早已放弃此连接，导致服务端资源被白白占用。这正是**半开连接**问题的根源。

三次握手中，服务端必须等到客户端回 ACK 才进入 ESTABLISHED。如果客户端不回，服务端超时后自动释放，不会产生资源泄漏。

三次握手保证了：

| 验证项 | 第几次完成 |
|--------|------------|
| 客户端发送能力正常 | 第一次 |
| 服务端接收能力正常 | 第一次 |
| 服务端发送能力正常 | 第二次 |
| 客户端接收能力正常 | 第三次（服务端在第三次 ACK 后确认）✅ |

### 状态变迁

```
Client:  CLOSED → SYN_SENT → ESTABLISHED
Server:  CLOSED → LISTEN → SYN_RCVD → ESTABLISHED
```

## 三、四次挥手：关闭连接

### 为什么需要四次？

TCP 是**全双工**协议——连接的两个方向相互独立。建立连接时，服务端可以把 SYN 和 ACK 合并成一个报文（因为收到 SYN 后立刻就能确认并建立连接）。但关闭连接时，两个方向必须分别关闭，且**服务端的 ACK 和 FIN 通常不能合并**——因为服务端收到客户端 FIN 后必须立刻回 ACK，但自己的 FIN 要等到剩余数据全部发送完才能发出，两者之间存在不确定的时间差。

### 过程

```
客户端 (Client)                          服务端 (Server)
  ESTABLISHED                            ESTABLISHED
     |                                        |
     |  1. FIN  SEQ=u                         |
     | -------------------------------------> |  → CLOSE_WAIT
     |  → FIN_WAIT_1                          |
     |                                        |
     |  2. ACK  ACK=u+1                       |
     | <------------------------------------- |
     |  → FIN_WAIT_2                          |  （发送剩余数据...）
     |                                        |
     |  3. FIN  SEQ=w, ACK=u+1               |  → LAST_ACK
     | <------------------------------------- |
     |  → TIME_WAIT                           |
     |                                        |
     |  4. ACK  ACK=w+1                       |
     | -------------------------------------> |  → CLOSED
     |                                        |
     |  等待 2MSL → CLOSED                    |
```

**第一次挥手（FIN）**：客户端发完所有数据，发送 FIN，进入 `FIN_WAIT_1`。此时服务端→客户端方向仍然开放，服务端可以继续发数据。

**第二次挥手（ACK）**：服务端立刻回 ACK，进入 `CLOSE_WAIT`（半关闭状态）。客户端收到后进入 `FIN_WAIT_2`，等待服务端的 FIN。

**第三次挥手（FIN）**：服务端把剩余数据发完，发送自己的 FIN，进入 `LAST_ACK`。

**第四次挥手（ACK）**：客户端回 ACK，进入 `TIME_WAIT`。服务端收到后立即 CLOSED。客户端等待 2MSL 后才真正 CLOSED。

### TIME_WAIT：为什么要等 2MSL？

`TIME_WAIT` 是四次挥手中最重要也最容易被忽视的状态。MSL（Maximum Segment Lifetime）是报文在网络中的最大存活时间，通常为 60 秒，2MSL 即 120 秒。

等待的原因有两个：

**1. 确保最后一个 ACK 能到达服务端**

若最后一个 ACK 丢失，服务端会超时重传 FIN。客户端在 TIME_WAIT 期间收到重传的 FIN，可以再次回 ACK。若客户端立即关闭，收到重传 FIN 时只能发 RST，服务端会报错，连接无法干净关闭。

**2. 让旧连接的残留报文在网络中消散**

2MSL 等于一个来回的最大传播时间。等待结束后，所有属于此连接的报文必然已消失。如果立即释放端口并建立新连接（相同的 IP+端口四元组），新连接可能收到旧连接的「幽灵报文」，造成数据错乱。

### 状态变迁

```
主动关闭方：ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
被动关闭方：ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
```

## 四、握手与挥手的对比

| 维度 | 三次握手 | 四次挥手 |
|------|---------|---------|
| 次数 | 3 次 | 4 次 |
| 目的 | 建立连接、同步 ISN | 关闭连接、各方独立结束 |
| 第二步能否合并 | 服务端 SYN+ACK 可合并 | ACK 和 FIN 通常不能合并 |
| 结束状态 | 双方 ESTABLISHED | 主动方经历 TIME_WAIT |
| 关键等待 | 无 | TIME_WAIT 等 2MSL |
| 对称性 | 非对称（服务端先 LISTEN） | 非对称（主动/被动关闭方不同） |

---

## 五、常见问题延伸

**CLOSE_WAIT 过多**：通常意味着服务端代码存在 bug——收到客户端 FIN 后，服务端没有主动调用 `close()`，导致连接永远卡在半关闭状态，无法进入 LAST_ACK，资源持续泄漏。

**TIME_WAIT 过多**：高并发短连接服务器（如 HTTP/1.0）中，大量 TIME_WAIT 会消耗端口资源。可通过 `SO_REUSEADDR`、`tcp_tw_reuse`（Linux）等内核参数缓解，或从架构层面改用长连接、连接池。

## 附录：探索路径

### Q1 — TCP 三次握手全过程

**问：** 解释 TCP 三次握手的过程

**答：** TCP 三次握手的核心目标是双方确认发送/接收能力正常，并同步初始序列号。第一次握手客户端发 SYN（SEQ=x）；第二次服务端回 SYN-ACK（SEQ=y, ACK=x+1）；第三次客户端回 ACK（ACK=y+1），双方进入 ESTABLISHED。两次握手不够的原因在于服务端无法确认客户端收到了自己的 ISN，且无法防止滞留的旧 SYN 报文造成半开连接资源浪费。

### Q2 — TCP 四次挥手全过程

**问：** 讲一下四次挥手

**答：** TCP 四次挥手因全双工特性必须分两个方向独立关闭。第一次客户端发 FIN；第二次服务端立即回 ACK（进入 CLOSE_WAIT）；第三次服务端发完剩余数据后发 FIN；第四次客户端回 ACK 后进入 TIME_WAIT，等 2MSL 后 CLOSED。挥手比握手多一次是因为服务端的 ACK 和 FIN 之间存在时间差（需要先发完剩余数据）无法合并。TIME_WAIT 等待 2MSL 的原因：一是保证最后一个 ACK 能被服务端收到（支持重传），二是让旧连接报文在网络中彻底消散，防止影响后续新连接。
