## Q1 TCP 三次握手全过程

**问：** 解释 TCP 三次握手的过程

**答：**

TCP 三次握手（Three-Way Handshake）是 TCP 协议在建立连接时使用的握手机制。它的核心目标是：**双方都确认自己的发送能力和接收能力正常，并同步初始序列号（ISN）**。

### 核心概念：序列号与标志位

TCP 报头中有几个关键字段：

- **SEQ（Sequence Number）**：当前报文的序列号
- **ACK（Acknowledgment Number）**：期望收到对方下一个字节的序列号
- **SYN 标志位**：用于连接建立，Synchronize 的缩写
- **ACK 标志位**：表示 ACK 字段有效

初始序列号（ISN）由操作系统随机生成，防止旧连接的报文干扰新连接。

---

### 三次握手过程

```
客户端 (Client)                          服务端 (Server)
     |                                        |
     |  1. SYN                                |
     |  SEQ=x, SYN=1                          |
     | -------------------------------------> |
     |                                        |  LISTEN 状态
     |                                        |  收到 SYN，进入 SYN_RCVD
     |                                        |
     |  2. SYN-ACK                            |
     |  SEQ=y, ACK=x+1, SYN=1, ACK=1         |
     | <------------------------------------- |
     |                                        |
     |  进入 SYN_SENT 收到 SYN-ACK            |
     |  进入 ESTABLISHED                       |
     |                                        |
     |  3. ACK                                |
     |  SEQ=x+1, ACK=y+1, ACK=1              |
     | -------------------------------------> |
     |                                        |  进入 ESTABLISHED
     |                                        |
  连接建立完成                             连接建立完成
```

---

### 三次握手详解

**第一次握手：Client → Server（SYN）**

- 客户端发送一个 SYN 报文
- 携带客户端的初始序列号 `ISN_c = x`
- 客户端进入 `SYN_SENT` 状态
- **意义**：告诉服务端「我想和你建立连接，我的初始序列号是 x」

**第二次握手：Server → Client（SYN + ACK）**

- 服务端收到 SYN 后，回复 SYN-ACK 报文
- `SEQ = y`（服务端自己的 ISN）
- `ACK = x + 1`（表示「已收到 x，期望下一个是 x+1」）
- 服务端进入 `SYN_RCVD` 状态
- **意义**：告诉客户端「我收到了你的连接请求，我的序列号是 y，我期望下一个收到 x+1」

**第三次握手：Client → Server（ACK）**

- 客户端收到 SYN-ACK 后，发送 ACK 报文
- `ACK = y + 1`（表示「已收到 y，期望下一个是 y+1」）
- 客户端进入 `ESTABLISHED` 状态
- 服务端收到后也进入 `ESTABLISHED` 状态
- **意义**：告诉服务端「我收到了你的序列号 y，连接可以开始了」

---

### 为什么是三次，不是两次？

两次握手存在一个致命问题：**服务端无法确认客户端有接收能力，也无法确认客户端收到了自己的 ISN**。

具体场景：

```
旧的 SYN 报文在网络中滞留 → 延迟到达服务端
服务端以为是新连接，回复 SYN-ACK
两次握手：服务端直接进入 ESTABLISHED，分配资源
但客户端早已放弃此连接 → 服务端资源被白白占用（半开连接攻击）

三次握手：客户端不回 ACK，服务端超时释放，不产生问题
```

三次握手保证了：

| 验证项 | 第几次握手完成 |
|--------|----------------|
| 客户端发送能力正常 | 第一次 |
| 服务端接收能力正常 | 第一次 |
| 服务端发送能力正常 | 第二次 |
| 客户端接收能力正常 | 第二次（但服务端不知道） |
| 服务端确认客户端接收正常 | 第三次 ✅ |

---

### 连接状态变迁总结

```
Client:  CLOSED → SYN_SENT → ESTABLISHED
Server:  CLOSED → LISTEN → SYN_RCVD → ESTABLISHED
```

💡 **未探索分支：**
- **SYN Flood 攻击**：攻击者大量发送 SYN 但不回第三次 ACK，耗尽服务端的半连接队列（SYN_RCVD 状态），导致正常连接无法建立——这是三次握手的经典安全问题。
- **四次挥手**：连接关闭时为什么需要四次，和握手有何本质区别？
- **ISN 随机化**：初始序列号为何要随机生成，不随机会有什么风险？

## Q2 TCP 四次挥手全过程

**问：** 讲一下四次挥手

**答：**

TCP 四次挥手（Four-Way Handshake）是 TCP 关闭连接的机制。与三次握手不同，**关闭连接需要四次，因为 TCP 是全双工的——两个方向的数据流必须分别独立关闭**。

---

### 核心标志位：FIN

- **FIN 标志位**：表示「我这边已经没有数据要发送了，请求关闭这个方向的连接」
- FIN 报文本身也占一个序列号（和 SYN 一样）

---

### 四次挥手过程

通常由客户端主动发起关闭（主动关闭方），服务端被动关闭。

```
客户端 (Client)                          服务端 (Server)
  ESTABLISHED                            ESTABLISHED
     |                                        |
     |  1. FIN                                |
     |  SEQ=u, FIN=1                          |
     | -------------------------------------> |
     |                                        |
     |  进入 FIN_WAIT_1                        |  收到 FIN，进入 CLOSE_WAIT
     |                                        |
     |  2. ACK                                |
     |  SEQ=v, ACK=u+1                        |
     | <------------------------------------- |
     |                                        |
     |  进入 FIN_WAIT_2                        |  服务端可继续发送剩余数据...
     |  （等待服务端的 FIN）                    |
     |                                        |
     |                                        |  数据发送完毕，进入 LAST_ACK
     |  3. FIN                                |
     |  SEQ=w, ACK=u+1, FIN=1                 |
     | <------------------------------------- |
     |                                        |
     |  进入 TIME_WAIT                         |
     |                                        |
     |  4. ACK                                |
     |  SEQ=u+1, ACK=w+1                      |
     | -------------------------------------> |
     |                                        |  收到 ACK，进入 CLOSED
     |                                        |
     |  等待 2MSL 后进入 CLOSED                |
  连接关闭完成                             连接关闭完成
```

---

### 四次挥手详解

**第一次挥手：Client → Server（FIN）**

- 客户端发完所有数据，发送 FIN 报文
- 进入 `FIN_WAIT_1` 状态
- **意义**：「我客户端没有数据要发了，请求关闭客户端→服务端这个方向」
- ⚠️ 此时服务端→客户端方向仍然开放，服务端还可以继续发数据

**第二次挥手：Server → Client（ACK）**

- 服务端收到 FIN，立刻回 ACK
- 服务端进入 `CLOSE_WAIT` 状态
- 客户端收到 ACK 后进入 `FIN_WAIT_2` 状态
- **意义**：「我知道你没数据了，但我这边可能还有数据没发完，稍等」

**第三次挥手：Server → Client（FIN）**

- 服务端把剩余数据发完后，发送自己的 FIN 报文
- 服务端进入 `LAST_ACK` 状态
- **意义**：「我也发完了，请求关闭服务端→客户端这个方向」

**第四次挥手：Client → Server（ACK）**

- 客户端收到服务端的 FIN，回 ACK
- 客户端进入 `TIME_WAIT` 状态，等待 **2MSL** 时间后才真正 CLOSED
- 服务端收到 ACK 后立即进入 `CLOSED`
- **意义**：「我收到了，连接正式关闭」

---

### 为什么是四次，不是三次？

握手时服务端可以把 SYN 和 ACK **合并**成一个报文，因为服务端收到 SYN 后马上就能确认并建立连接。

挥手时**不能合并**第二次和第三次：

```
第二次（ACK）：服务端必须立刻回复，告诉客户端「我知道你关了」
               ↓
第三次（FIN）：服务端需要等自己发完剩余数据才能关闭，时机不确定

两者之间存在时间差 → 必须分开发送 → 所以是四次
```

如果服务端没有剩余数据，理论上可以把 ACK 和 FIN 合并，退化为三次挥手——这在实际中确实存在，叫做**同时关闭（Simultaneous Close）**或 TCP 延迟 ACK 优化的特例。

---

### TIME_WAIT 状态：为什么要等 2MSL？

`TIME_WAIT` 是四次挥手中最容易被问到的考点。客户端发完最后一个 ACK 后，**不立即关闭，而是等待 2 个 MSL（Maximum Segment Lifetime，报文最大生存时间，通常为 60s）**。

原因有两个：

**原因一：确保最后一个 ACK 能到达服务端**

```
客户端发出 ACK
        ↓
如果 ACK 丢失，服务端会重传 FIN（超时重传）
        ↓
客户端在 TIME_WAIT 期间收到重传的 FIN，可以再次回 ACK
        ↓
如果客户端立即 CLOSED，收到重传 FIN 会发 RST，导致服务端报错
```

**原因二：让网络中残留的旧报文消失**

```
2MSL = 一个来回的最大存活时间
等待 2MSL 后，所有属于此连接的报文都已在网络中消失
新建立的同 IP+端口 连接不会收到旧连接的「幽灵报文」
```

---

### 连接状态变迁总结

```
主动关闭方（Client）：
ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED

被动关闭方（Server）：
ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
```

---

### 握手 vs 挥手对比

| 维度 | 三次握手 | 四次挥手 |
|------|---------|---------|
| 次数 | 3 次 | 4 次 |
| 目的 | 建立连接、同步 ISN | 关闭连接、各方独立结束 |
| 能否合并 | 服务端 SYN+ACK 可合并 | 第 2、3 次通常不能合并 |
| 结束状态 | 双方 ESTABLISHED | 主动方 TIME_WAIT → CLOSED |
| 关键等待 | 无 | TIME_WAIT 等 2MSL |

💡 **未探索分支：**
- **TIME_WAIT 过多的问题**：高并发服务器上大量 TIME_WAIT 会耗尽端口资源，如何用 `SO_REUSEADDR`、`tcp_tw_reuse` 等手段缓解？
- **CLOSE_WAIT 过多的问题**：通常意味着服务端代码 bug——收到 FIN 后没有主动调用 close()，连接卡在半关闭状态。
